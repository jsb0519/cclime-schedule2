# 뷰티짱 대조: 근무 외 상태 전체 + 반차 부분금지 3단계 대조 (설계)

- 날짜: 2026-07-22
- 브랜치: `feature/beautyzzang-offday-levels`
- 담당: 장상범
- 선행: [[2026-07-22-beautyzzang-schedule-compare-design]] (1차 대조 기능 — 종일휴무만)

## 배경 / 문제

기존 대조 기능은 근무표에서 **`휴무`·`휴직`만** 뷰티짱 휴무와 대조했다. 그런데:

1. **근무 외 종일 상태가 누락**: `연차·대휴·교육`도 종일 근무 안 하는 날인데 대조 기준에서 빠져,
   뷰티짱에 종일휴무로 등록돼 있으면 **빨강 오탐**이 뜬다.
   (규칙: "근무표에 등록된 근무 외 모든 건 뷰티짱 휴무로 이해한다" — 장상범)
2. **반차 미대조**: 근무표 `오전반차·오후반차`는 뷰티짱에 **시간대 부분금지**(`n1DayHoliday=0`,
   사유 "기타"/"휴무")로 등록되는데, 봇이 종일휴무(`=1`)만 가져와서 아예 대조되지 않는다.

## 원천 데이터 사실 (RECON 기준)

뷰티짱 BanList(`StatusBoardV2-AjaxReservationBanList`) row:
- `n1DayHoliday=1` → **종일 휴무**. (봇이 현재 수집)
- `n1DayHoliday=0` → **시간대 부분금지**(`strTimeS`~`strTimeE`, 사유 "기타"/"휴무" 등). (현재 미수집)
- 부분금지는 routine(매일 점심 등)이 아니라 ad-hoc(`rv_ban` 예약금지). 픽스처 실측: 종일 8 : 부분 1.
  → 부분금지를 대조에 넣어도 대량 오탐 위험 낮음.

## 결정된 요구사항

- **A안 채택**(3단계 레벨). 오전/오후 구분 없이 "부분금지 유무"만 본다.
- 종일 오프 5종(휴무·연차·대휴·교육·휴직)은 무조건 종일 대조 기준에 포함.

## 비교 모델 — 3단계 레벨

각 (디자이너, 날짜)를 양쪽에서 레벨로 매긴다:

| 레벨 | 근무표 (day_types / off_days) | 뷰티짱 (BanList) |
|---|---|---|
| **2 = 종일 오프** | 휴무 · 연차 · 대휴 · 교육 · 휴직 | `n1DayHoliday=1` |
| **1 = 반일 오프** | 오전반차 · 오후반차 | `n1DayHoliday=0` (그날 부분금지 1건 이상) |
| **0 = 근무** | 근무 (기본값) | 해당 날짜 레코드 없음 |

**대조 규칙** (기존 색 의미의 자연스러운 일반화):

- 뷰티짱 레벨 **<** 근무표 레벨 → 🟠 **주황**: 근무표는 (더) 쉬는데 뷰티짱이 덜 막음 = **예약 들어올 위험**(핵심 신호).
- 뷰티짱 레벨 **>** 근무표 레벨 → 🔴 **빨강**: 근무표는 (더) 근무인데 뷰티짱이 더 막음 = 예약 손실/데이터 불일치.
- 같으면 → 일치(무표시).

> 기존 코드와의 호환: 기존은 2단계였고 red=`inBz && !inSch`(뷰티짱off·근무표work=레벨 2>0),
> orange=`!inBz && inSch`(근무표off·뷰티짱none=레벨 0<2). 3단계 레벨 비교는 이 두 경우를 그대로 포함한다.

**예시:**
- 근무표 `연차`(2) × 뷰티짱 종일휴무(2) → **일치** (기존엔 빨강 오탐이던 것 해소).
- 근무표 `오전반차`(1) × 뷰티짱 부분금지 있음(1) → **일치**.
- 근무표 `오전반차`(1) × 뷰티짱 없음(0) → 🟠 주황 (반차 미등록).
- 근무표 `근무`(0) × 뷰티짱 부분금지(1) → 🔴 빨강 (드묾).
- 근무표 `휴무`(2) × 뷰티짱 부분금지만(1) → 🟠 주황 (종일 쉬는데 반나절만 막힘).

## 컴포넌트별 변경

### 1) 봇 — `bz_sync/`

- **`banparse.parse_offdays`** (이름 유지 또는 `parse_bans`로): row를 종일/부분으로 분류.
  반환 구조를 `{oidStaff: {"off": [YYYY-MM-DD], "half": [YYYY-MM-DD]}}` 로 변경.
  - `n1DayHoliday==1` → `off`
  - `n1DayHoliday==0` → `half` (같은 날 여러 부분금지는 dedupe해 날짜 1개로)
  - 기존처럼 `oidStaff`/`strDate` 없는 row skip.
- **`names.resolve`**: off/half 각각의 oid→이름 매핑 적용(동일 로직 2회 또는 dict 순회).
  system_accounts 제외·name_map 적용은 양쪽 동일.
- **`fb.build_payload`**: `{name: {"off":[...], "half":[...]}}` 형태 + `_syncedAt` 생성.
  빈 배열은 생략하지 않고 유지(리더가 `off||[]`, `half||[]`로 방어).
- **`fb.write_branch`/키 규칙**: 변경 없음(`fb_key`, 경로 동일).
- **`sync.py`**: 반환 구조 변경에 맞춰 파이프 연결만 조정. 월범위·재시도·오류처리 로직 불변.

**Firebase 노드 구조 (신규):**
```json
"2026_07": {
  "01_ 신사본점": {
    "_syncedAt": "2026-07-22T…+09:00",
    "김보경": { "off": ["2026-07-03","2026-07-10"], "half": ["2026-07-17"] },
    "…": { "off": [...], "half": [] }
  }
}
```

### 2) 앱 — `index.html`

- **`_schOffMap` 생성부(현 2659~2663)**: 단일 Set → 레벨 분리 + 매칭여부 분리.
  - full 세트: `dtype ∈ {휴무,연차,대휴,교육,휴직}`
  - half 세트: `dtype ∈ {오전반차,오후반차}`
  - 결과를 `{name: {full:Set, half:Set}}` 로 `buildBeautyzzangTable`에 전달.
  - **`_schNames` Set 추가**: 근무표 렌더 루프가 도는 **모든 직원 이름**을 담는다(오프 유무 무관).
    현재는 오프 있는 직원만 `_schOffMap`에 생겨, 매일 근무 직원이 `(미매칭)`으로 오표기되는
    잠재버그가 있다 → `_schNames`로 "근무표에 존재하는가"와 "그날 오프인가"를 분리한다.
  - (참고: `DAY_TYPES_CYCLE`·`TYPE_CLS`에 이미 위 타입들 존재. `offCnt` 카운트는 현행 유지.)
- **`buildBeautyzzangTable` 리더/병합(현 2467~2493)**: 노드값이 `{off,half}` 객체.
  - 강남 등 병합지점: 이름별 `off`끼리, `half`끼리 각각 합집합.
  - **방어**: 노드값이 배열(구형)이면 `{off: 배열, half: []}`로 해석.
- **판정/렌더(현 2508~2527)**: 3레벨 비교로 교체.
  - `bzLv` = 종일 2 / 부분 1 / 없음 0. `schLv` = full 2 / half 1 / 근무 0.
  - `bzLv < schLv` → 주황, `bzLv > schLv` → 빨강, 같으면 무표시.
  - 마크: 뷰티짱 종일=`휴`, 부분=`반`, (주황이고 뷰티짱 표시없음)=`·`.
  - **매칭 판정은 `_schNames.has(name)` 로**: 이름이 근무표에 있으면 매칭(오프 0건이어도 레벨0으로 비교),
    없으면 `(미매칭)` 라벨 + 하이라이트 없이 뷰티짱 데이터만 표시(현행 유지).
- **범례**: 그리드 상단 요약 옆에 `휴=종일 · 반=부분 · 🟠근무표 오프 미반영 · 🔴뷰티짱 과다차단` 한 줄.

### 3) 테스트 — `bz_sync/tests/`

- 픽스처에 `n1DayHoliday=0` 부분금지 row 포함 케이스 추가(신규 또는 기존 확장).
- `parse_offdays`: off/half 분리 검증(같은 날 종일+부분 동시, 부분만, 종일만, 다중 부분금지 dedupe).
- `build_payload`: `{name:{off,half}}` 구조 검증.
- `names.resolve`: off/half 양쪽 매핑·system 제외 검증.
- 기존 16 테스트 회귀 통과 유지.
- **앱 JS는 자동 테스트 없음** → Playwright 로컬 서버로 강남 병합지점 육안 검증(현행 절차 재사용).

### 4) 재동기화 (사람이 직접)

노드 구조가 바뀌므로 배포 후 **사람이 봇 재실행** 필수(Claude 실행은 안전분류기에 막힘):
`cd ~/github/cclime-schedule2 && python3 -u -m bz_sync.sync bz_sync/config.json`
기대 `전체 성공 (20지점)`. 이후 `off/half` 구조로 노드 갱신 확인.

## 하위호환 / 마이그레이션

- 리더가 구형 flat 배열을 `{off:배열, half:[]}`로 방어 해석하므로, 재동기화 전에도 앱이 깨지지 않는다.
- 재동기화하면 언더스코어 키 노드가 신구조로 덮인다. 옛 고아 키(언더스코어 없는)는 무관(그대로 방치).

## 비목표 (YAGNI)

- 오전/오후 반차의 시간대 정밀 매칭(부분금지 유무만 본다).
- 부분금지 사유별(병가/기타) 세분화.
- 뷰티짱→근무표 역방향 자동수정. (대조·표시만)
- 지점별 이름매핑(동명이인 박예진 등)은 별도 과제로 유지.

## 리스크

- 부분금지가 반차 아닌 개인용무(예: 저녁 시간 블록)인 경우 "근무 × 부분금지=빨강"이 뜰 수 있음.
  드물다고 판단(픽스처 1건)하나, 실데이터에서 노이즈 과다 시 "근무×부분금지는 무시" 옵션 재검토.
