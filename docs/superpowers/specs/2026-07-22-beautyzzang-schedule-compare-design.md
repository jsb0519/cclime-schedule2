# 뷰티짱 휴무 ↔ 근무표 대조 기능 — 설계 문서

- 날짜: 2026-07-22
- 담당: 장상범
- 대상 앱: cclime-schedule2 (근무표) — https://jsb0519.github.io/cclime-schedule2/
- 외부 시스템: 뷰티짱(헤어짱칵테일 / HASYS CRM) — https://hasys.hairzzang.com

## 1. 목적

각 지점의 **근무표(cclime-schedule2)**와 뷰티짱 CRM에 등록된 **직원 휴무 스케줄**이
일치하는지 한눈에 대조한다. 한 지점을 선택하면 **위에는 근무표 그리드**, **그 아래에
뷰티짱 휴무 그리드를 동일 양식(직원 × 날짜)**으로 표시하고, 두 데이터의 **불일치를
하이라이트**한다.

## 2. 배경: 시스템 현황 (2026-07-22 실사 기준)

### 2.1 근무표 앱 (cclime-schedule2)
- 정적 HTML(`index.html`) + GitHub Pages 배포.
- 직원 데이터: Google Sheet(GViz JSONP).
- 스케줄 데이터: Firebase Realtime DB(REST).
  - `FB_BASE = https://cclime-schedule-cb047-default-rtdb.asia-southeast1.firebasedatabase.app`
  - `/schedule/YYYY_MM/{지점}` , `/overflow/YYYY_MM`
- 지점 20개. 지점 선택 드롭다운으로 월간 그리드(직원 행 × 날짜 열) 렌더.
- 상태 코드: 근무 / 휴무 / 휴직 / 반차 / 연차.

### 2.2 뷰티짱 CRM (HASYS)
- 레거시 **ASP.NET WebForms**(`__VIEWSTATE`, `__EVENTVALIDATION`, doPostBack), IE모드 지향.
- 프레임셋 구조: `frame_main`(앱 콘텐츠) / `frame_config` / `frame_push`(`Library/Api/Controller/Push.aspx`).
  - **`frame_push`가 지속 연결(long-poll)** → 헤드리스 브라우저가 "로드 완료"에 도달하지 못하고 멈춤.
  - **대응: 프레임셋 통째 로딩 대신, 로그인 후 콘텐츠 페이지/AJAX 엔드포인트에 직접 접근.**
- 로그인: 폼 방식(샵코드 / 아이디 / 비밀번호), 세션 쿠키 기반. 본사 계정은 지점 전환 가능(우상단 선택: 본사 → 지점).
- **엑셀 내보내기 존재**: `fnExcelDownloadAuth()` → `/include/Common/ExcelDownlaodAuth.aspx`(팝업 인증).

### 2.3 뷰티짱 휴무 데이터 소스 후보 (실사 확인)
| 화면 | 콘텐츠/엔드포인트 | 데이터 형태 | 비고 |
|---|---|---|---|
| 근태현황 | `POST /CRM.staff/Diligence-ListByDiligenceAjaxList` | 담당자별 월간 **집계**(근무·결근·휴무일수…) | `ddlOfficeGubun` 현지점/전지점 有. 날짜별 아님(개수만) |
| 휴무캘린더 | `POST /CRM.Regist/Reservation-HolidayCalendar` | 달력에 휴무 표시 | 날짜별 |
| 예약현황판(StatusBoardV2) | `/CRM.Reservation/StatusBoardV2` | 디자이너별 **당일** 스케줄+OFF | 상범님이 실제 보는 화면. 날짜별 순회 필요 |
| 근태입력 | `/staff/diligence/default.aspx?page=Diligence` | 담당자 × 날짜 입력 그리드 | 등록 화면. 월 그리드 형태 가능성 |

> **정본 소스는 구현 초반에 실제 데이터를 열어 확정한다.** 근무표가 날짜별 그리드이므로
> **직원별·날짜별 휴무**가 필요하다. 후보 중 "월 1회 요청으로 지점 전체 그리드"를 주는
> 화면(근태입력 / 휴무캘린더)을 우선 검토하고, 예약현황판은 대조·백업용으로 둔다.

## 3. 전체 아키텍처

```
[맥미니] cron/launchd (매일 새벽 1회)
   └─ sync-bot (Node + Playwright, 세션 유지)
        · 뷰티짱 로그인 (creds = 맥미니 로컬 파일, repo에 없음)
        · 20개 지점 순회: 지점 전환 → 월간 휴무 화면 로드 → 표 파싱
        · 결과를 Firebase에 기록 (당월 + 익월)
             ↓ write
[Firebase RTDB]  /beautyzzang/YYYY_MM/{지점} = { 직원명: [휴무일…], _syncedAt }
             ↓ read
[근무표 앱 index.html]
   · 지점 근무표 그리드 (기존)
   · 그 아래 "뷰티짱 휴무" 그리드 — 동일 양식(직원 × 날짜)
   · 두 그리드 대조 → 불일치 셀 하이라이트 + 요약
```

## 4. 컴포넌트 설계

### 4.1 Sync bot (맥미니, 신규)
- **런타임**: Node.js + Playwright(실제 Chromium). 레거시 JS·세션 처리 위해 실브라우저 사용.
- **동작**:
  1. 로그인(폼 제출) → 세션 확보.
  2. `frame_push`가 물린 프레임셋은 피하고, **콘텐츠 페이지에 직접 접근**해 표 DOM만 읽는다.
  3. 지점 20개 순회(지점 전환 메커니즘은 구현 시 확정), 당월+익월 휴무 파싱.
  4. `{ 지점: { 직원명: ["YYYY-MM-DD", ...] } }` 구성.
  5. Firebase `/beautyzzang/YYYY_MM/{지점}`에 write. `_syncedAt`(KST) 기록.
- **입출력 계약**:
  - 입력: 없음(스케줄 실행). 설정은 로컬 config(지점 목록·명 매핑·creds 경로).
  - 출력: Firebase write + 실행 로그. 실패 시 알림.
- **의존성**: 뷰티짱 세션, Firebase REST, 로컬 creds 파일.
- **실패 처리**: 로그인/파싱 실패 시 **이전 데이터 유지**(덮어쓰지 않음) + **알림(텔레그램 노바 또는 슬랙)**. 조용한 실패 금지.

### 4.2 직원명 매핑 (신규, 소량 데이터)
- 뷰티짱 담당자명 ≠ 근무표 직원명일 수 있고, 시스템 계정(결제변경·워크인·소멸·당일취소 등)이 섞임.
- **필터 규칙**(시스템 계정 제외) + **명 매핑 테이블**(뷰티짱명 → 근무표명)을 config로 둔다.
- 매핑 실패(어느 쪽에만 존재하는 이름)는 대조 UI에서 "미매칭"으로 별도 표시.

### 4.3 Firebase 스키마 (신규 네임스페이스)
```
/beautyzzang/2026_08/01. 신사본점 = {
  "홍길동": ["2026-08-03","2026-08-17"],
  "_syncedAt": "2026-08-01T01:00:00+09:00"
}
```
- 기존 `/schedule`, `/overflow`는 **변경하지 않는다**(읽기 전용 대조).
- 지점 키는 근무표의 지점 표기와 동일하게 맞춘다(예: `01. 신사본점`).

### 4.4 프론트(근무표 앱 `index.html`, 수정)
- 지점 선택 시, 근무표 그리드 렌더 직후 **뷰티짱 휴무 그리드**를 같은 날짜 열 정렬로 아래에 추가.
- 데이터 로드: `GET {FB_BASE}/beautyzzang/{YYYY_MM}/{지점}.json`.
- **대조 로직**: 직원·날짜별로 (근무표 상태) vs (뷰티짱 휴무 여부) 비교.
  - 근무표 "휴무/휴직" ↔ 뷰티짱 휴무: 일치.
  - 근무표 "근무"인데 뷰티짱 휴무 → 불일치(빨강).
  - 뷰티짱 휴무 없는데 근무표 "휴무" → 불일치(주황).
  - 미매칭 직원 → 회색/별도 라벨.
- 상단 요약: "N건 불일치", "마지막 동기화: {_syncedAt}".
- 데이터 없음/오래됨 시 안내 문구.

## 5. 데이터 흐름
1. (매일) 맥미니 봇 → 뷰티짱 스크레이프 → Firebase `/beautyzzang` write.
2. (조회) 사용자 → 근무표 앱 지점 선택 → `/schedule`(기존) + `/beautyzzang`(신규) 동시 read → 두 그리드 + 대조 렌더.

## 6. 에러 처리 / 신뢰성
- 봇: 로그인 실패, 지점 전환 실패, 표 구조 변경 → 각 단계 예외 잡고 **부분 성공 허용**(성공 지점만 갱신), 실패 지점은 이전 값 유지 + 알림.
- 레거시 스크레이핑은 **주기적으로 깨질 것을 전제** → 실패 알림 + 앱/봇에 **수동 트리거(백업 경로)** 병행 권장.
- 프론트: Firebase 미응답·데이터 결측 시 근무표는 정상 표시하고 뷰티짱 영역만 "동기화 데이터 없음" 표시.

## 7. 보안
- 뷰티짱 비밀번호는 **맥미니 로컬 파일에만**(기존 `hr_credentials.json` 패턴 준용). repo·채팅·로그 노출 금지.
- 봇 로그에 자격증명 값 출력 금지(성공/실패만).
- Firebase 규칙: 현재 공개 read/write는 별도 개선 이슈로 남김(본 기능 범위 밖, 기존 리스크).

## 8. 테스트 전략
- **봇 단위**: 파서에 저장된 HTML 픽스처를 넣어 `{직원: [휴무일]}` 추출 검증(뷰티짱 접속 없이).
- **명 매핑**: 시스템 계정 필터·매핑 테이블 단위 테스트.
- **대조 로직**: 근무표/뷰티짱 목업으로 일치·불일치·미매칭 케이스 검증.
- **통합(수동)**: 한 지점(신사점) 실데이터로 그리드·하이라이트 육안 확인.

## 9. 범위 밖 (YAGNI)
- 뷰티짱으로의 **역방향 쓰기**(휴무 등록/수정) — 대조만, 쓰기 없음.
- 전 지점 자동 정합성 리포트/알림(1차 범위 밖, 추후).
- Firebase 보안 규칙 개편(별도 이슈).

## 10. 열린 항목 (구현 초반 확정)
1. 정본 휴무 소스 화면 확정(근태입력 / 휴무캘린더 / 예약현황판) — 실데이터 확인.
2. 지점 전환 메커니즘(우상단 선택의 실제 요청/파라미터).
3. 봇 실행 방식(launchd vs cron) 및 실패 알림 채널(노바/슬랙).
4. 직원명 매핑 초기 테이블 작성.
